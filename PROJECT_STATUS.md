# 项目状态

本文件是持续 AI 协作维护时的简洁项目记忆入口。

## 当前质量门禁

- 仓库卫生检查：`python tools/check_repository_hygiene.py`
- 后端 lint 门禁：`black --check .`、`isort --check-only .`、`ruff check app/ --select E9,F63,F7,F82`
- 后端测试：`python -m pytest -q`
- Python 依赖审计：`python -m pip_audit -r requirements.txt -r requirements-dev.txt --strict`
- 前端门禁：`npm run typecheck`、`npm run build`、`npm run audit:security`
- Runtime 门禁：`dotnet build --configuration Release --no-restore`、`dotnet test --configuration Release --no-build`

## 当前架构事实

- 根目录 `.github/workflows/ci.yml` 是唯一权威 CI workflow。
- 根目录 `VERSION` 是项目默认版本源；README 徽章和后端 `settings.VERSION` 默认值必须与它一致。
- 前端 `package.json` 和 `package-lock.json` 根包名必须是 `dbooth-frontend`，版本必须与根 `VERSION` 一致。
- 前端设置页“关于”信息必须展示真实前端版本，不得硬编码与根版本源冲突的构建号、设备 ID 或授权状态。
- 不得新增 `D-Booth/backend/VERSION`、`D-Booth/backend/app/services/VERSION`、`D-Booth/frontend/VERSION` 或 `D-Booth/runtime-dotnet/VERSION` 等额外版本源。
- 仓库卫生检查同时覆盖已跟踪和未跟踪的生成型报告/本地产物。
- 前端依赖管理使用 npm 和 `package-lock.json`。
- 前端文档、部署示例和开发指南不得回退到 pnpm/yarn。
- 前端是应用而不是库，`react` 和 `react-dom` 必须作为直接 runtime dependencies 声明。
- 前端 React runtime 与 `@types/react` / `@types/react-dom` 必须保持同一主版本。
- 后端 pytest 配置集中在 `D-Booth/backend/pyproject.toml`。
- 后端全量 mypy 不是 CI 门禁；当前 CI 使用 Ruff 致命错误检查，类型债务按模块逐步收敛。
- 后端模型 UUID 类型位于 `app.models.custom_types`；不要重新创建 `app.models.types`。
- Docker 发布上下文必须具备 `D-Booth/backend/Dockerfile` 和 `D-Booth/frontend/Dockerfile`。
- Docker 发布必须等待后端、前端、Runtime 和安全扫描全部通过，镜像命名空间来自 `DOCKER_USERNAME` secret。
- GitHub Release 由 release job 使用 `gh release create` 创建，不使用旧 release action；没有真实部署脚本时，CI 不应声明 staging/production 部署成功。
- CI 中第三方 GitHub Actions 必须使用版本 tag，不能使用浮动 `main` 或 `master` 分支。
- CI 安全扫描中的 Python 依赖审计必须使用项目声明的 `PYTHON_VERSION`，不能依赖 runner 默认 Python。
- Green Screen 设置和背景资产由后端数据库持久化；前端页面已接入读取、保存、背景上传、删除、预览和测试照片分析接口，并通过统一 API 地址解析后端上传资产。
- Booth 管理页必须通过真实 `team_id` 调用后端 `/booths`，不得提供后端 schema 不支持的假锁定成功操作。
- 虚拟助手前端播放列表、TTS 播放和设置页试听必须通过统一 API 地址访问后端；设置页试听使用当前编辑文本。
- 离线队列同步必须复用统一 API 客户端；打印任务使用后端真实 `/print-jobs` 接口，不得调用不存在的 `/api/v1/print`。
- 前端 Web Vitals 上报必须显式配置 `VITE_WEB_VITALS_ENDPOINT`；未配置时只保留开发环境 console 输出，不得默认请求后端不存在的 analytics endpoint。
- 前端统一 API 客户端必须同时支持调用方取消信号和内部 timeout；调用方取消应保持 `AbortError` 语义，不得误包装为 timeout 或普通请求失败。
- 分享设置 WhatsApp 号码字段的权威名称是 `whatsapp_number`；后端只应输出正确字段，但需要继续读取历史拼错的 `whatssapp_number` 配置。
- 后端本地文件访问或删除的上传目录边界必须使用 `Path.relative_to()` 这类真实路径归属判断，不得用字符串 `startswith` 判断路径前缀。
- Green Screen 本地背景/叠层资产的可访问 URL 使用 `/api/v1/green-screen/assets/{event_id}/{folder}/{filename}`；前端需要继续兼容历史 `/uploads/green-screen/...` URL，但不得公开整个 `/uploads` 目录。
- Green Screen 设置、背景上传/删除和批处理请求都必须验证当前用户已认证，并且是对应活动所属团队成员。
- 调查和免责声明的管理配置写入、调查回答导出必须验证当前用户已认证，并且是对应活动所属团队成员；现场来宾读取配置和提交/接受流程保持独立处理。
- 同步接口 `/sync/state`、`/sync/push`、`/sync/pull`、`/sync/log` 必须验证当前用户已认证，并且是对应 booth/team 的团队成员。
- Booth 管理接口必须验证当前用户是 booth 所属团队成员；注册 booth 时必须验证目标团队成员身份，且 `current_event_id` 必须属于同一团队。
- 活动管理页不得保留“功能开发中”的新建活动入口；创建活动应调用后端 `/events`，团队缺失时明确提示，状态筛选按钮必须真实过滤列表。
- 活动列表不得保留仅弹提示的设置操作；未接入活动配置页前，只展示真实可用的详情和进入拍照操作。
- 活动管理页不得在未登录、接口失败或后端无活动时回退到可操作的演示活动；静态汇总、假最近活动和假收入图表不得替代真实数据或空状态。
- 工作台不得提供绕过活动会话的直接拍照入口；开始拍摄应进入活动管理选择真实活动，首页不得展示硬编码设备在线、趋势图或运营统计。
- 数据统计页不得用静态业绩、固定增长率、假趋势或假活动分布替代后端统计；后端只提供汇总时，趋势/分布应显示真实空状态或能力说明。
- 前端共享常量文件不得保留未引用的演示活动、假统计、假分享方式、假打印机或打印预览占位数据；可视化占位素材只能服务真实空状态或编辑预览。
- GoPro 页面不得展示固定演示设备或用本地定时器模拟扫描/连接成功；发现、连接、状态读取和断开连接必须调用后端 `/gopro` 真实接口。
- 系统设置页未接入真实配置或账户管理流程的项目只能作为只读状态展示，不得提供“开发中”点击入口。
- 系统设置页不得展示未接入后端的假订阅到期、假 AI 积分或假最近登录时间；这些账号/订阅状态必须来自真实账户系统，否则只能显示未接入说明。
- 锁屏不得内置或接受公开默认 PIN；未配置 PIN 或仍使用 `1234` 这类演示 PIN 时必须阻止解锁，而不是给现场设备提供可猜密码。
- 系统设置页必须提供锁屏 PIN 配置入口；PIN 必须为 4-6 位数字且不得为 `1234`，已有 PIN 时修改需要输入当前 PIN，设置页不得展示当前 PIN 明文。
- 相册页不得用“开发中”提示代替下载/筛选操作；本地可展示照片至少应支持浏览器下载，筛选按钮必须真实改变照片列表。
- 分享页不得生成本地假分享链接、演示邮件/短信或只记录渠道后提示成功；分享必须基于已上传后端照片和真实分享链接，未接入的 AirDrop、微信、云端相册等渠道不得展示为可点击入口。
- 分享页不得展示默认示例照片或静态假统计；无当前照片时必须展示空状态，状态数据应来自当前会话真实状态。
- 虚拟助手播放列表读取和按 timing 获取 TTS 音频是现场播放链路，保持公开读取；播放列表写入必须验证当前用户已认证且是活动所属团队成员。
- 虚拟助手 TTS 试听接口必须验证当前用户已认证；当前试听请求不携带 `event_id`，因此不做团队归属校验。
- 打印机枚举、状态、队列、取消任务、测试页、校准和 Booth 聚合硬件健康接口必须验证当前用户已认证。
- 签名上传是现场采集链路，保持公开但必须验证目标 photo session 存在；签名读取和删除必须验证当前用户已认证且是 session 所属活动团队成员。
- 问卷提交和免责声明接受是现场采集链路，保持公开但必须验证 `session_id` 属于目标 `event_id`；问卷答案按 session 读取必须验证当前用户已认证且是 session 所属活动团队成员。
- 美颜状态和预设是公开能力/静态配置；美颜预览、人脸检测和异步美颜任务提交必须验证当前用户已认证。
- 前端美颜页必须在打印或分享前将已处理的美颜 Blob 写回 `useCaptureFlow` 当前照片流；后续打印/分享不得继续使用处理前原图。
- 前端美颜页贴纸必须同时在预览层正确显示并在打印/分享前合成为输出 Blob；默认 emoji 贴纸不能用 `<img src>` 渲染。
- 前端美颜页只应暴露已经能进入打印/分享输出闭环的工具；滤镜、调色、文字、裁剪等工具必须完成真实预览和输出合成后再开放给顾客。
- 前端美颜页读取后端返回的相对媒体 URL 时必须通过统一后端地址解析，避免跨源部署下在前端域名请求上传资产。
- AI 功能中心只应展示已接入真实页面或闭环能力的入口；未接入的 AI 视频、证件照、头像、海报、品牌生成、积分充值等不得作为可点击入口展示。
- 绿幕本地资产读取保持公开以支持前端图片展示；绿幕预览、测试照片分析、设置、背景上传/删除和批处理请求必须验证当前用户已认证。
- 模板中心和编辑器使用后端 `/templates` CRUD；团队选择优先当前活动团队，缺失时回退到登录用户第一个团队，保存到后端和导出 JSON 是两个独立动作。
- 模板自定义创建优先提供实用打印版式快速起步；模板中心和编辑器空画布都应能直接选择常用版式。基础版式参考 `D:\安装包归档\单机离线版免安装2.0版本\content\Templates` 的 300DPI 坐标，但不依赖外部资源文件。
- 模板中心的静态推荐/热门卡片必须接入可编辑的快速版式创建链路，不得用“使用”入口把顾客带到空白模板编辑器。
- 前端拍照/打印流程的当前模板状态由 `useCaptureFlow` 管理；模板中心“使用”、模板编辑器保存、相机页模板提示和打印预览页渲染必须使用同一状态源。
- 从相机页或打印预览页进入模板中心时属于模板选择模式，快速/推荐/热门版式必须能直接设为当前打印模板并返回原页面；只有模板管理入口才默认进入编辑器创建。
- 模板选择模式下的快速/推荐/热门版式用于真实打印前必须保存成后端模板，并以真实 UUID 作为当前模板 ID；`quick-*` 这类前端临时 ID 只能作为未登录本地预览态，不得提交给 `/print-jobs`。
- 从打印预览页进入模板中心选择模板时，模板中心“使用”必须返回打印预览页；从相机页进入模板中心时仍返回相机页，避免顾客更换模板后脱离当前打印确认流程。
- 相机拍照后的打印主动作应进入打印预览而不是暗示立即出纸；拍后操作区必须保留选择/更换模板入口，让顾客能按“拍照 -> 选模板 -> 预览 -> 打印”跑通。
- 相机拍照后的美颜、选择/更换模板和打印预览操作不能只跟随短暂闪光动画显示；拍后决策入口应持续可见，直到顾客选择继续动作或重拍。
- 相机页“重拍”必须移除刚拍照片并释放本地 blob URL，避免坏照片继续占用当前模板照片槽位。
- 相机页和打印预览页必须按当前模板照片槽位显示“需拍/已拍”状态；未选模板时主动作进入模板中心，模板照片不足时主动作回到拍照补齐，不得让顾客把未完成模板当作最终成品。
- 打印预览页必须在主预览区域和打印设置区域展示同一套模板照片槽位状态；缺照片时应提供直接返回相机补拍的操作。
- 相机页最近照片栏只能展示当前会话真实照片；无照片时必须展示空状态，不得用静态样张伪装为最近拍摄。
- 相机权限未开启、浏览器不支持相机或视频帧未就绪时不得把示例图加入当前拍摄流；示例图只能作为视觉占位，不能进入后续上传、模板预览、打印或分享链路。
- DSLR 拍摄结果只有创建后端 `Photo` 并写入 `useCaptureFlow` 当前照片流后，才算进入顾客打印/分享闭环；仅返回本地 `local_path` 不得被视为可打印照片。
- 前端拍摄流展示后端返回的相对媒体 URL 时必须通过统一后端地址解析，避免跨源部署下在前端域名请求 `/uploads/...`。
- 后台照片处理、拼图、美颜等任务在未配置 R2 时如果返回 `/uploads/...`，必须先真实写入对应本地文件；不得返回不存在的本地 URL 作为成功结果。
- 打印预览页在已有照片但未选模板时必须把模板选择作为明确待办展示，不应让未套模板的临时预览看起来像最终成品。
- 当前打印预览和后端真实打印都按前端模板 layout 渲染照片框、文本、形状、图片/二维码/日期占位；打印任务携带 `template_id`，执行时以主照片为第一个照片框，并从同一拍摄 session 按创建顺序补齐其它照片框。
- 多照片模板不得在缺失后续照片时用第一张照片回填空槽位；前端预览应显示缺失槽位，后端创建打印任务必须校验同一 session 的照片数量满足模板最大 `photoNumber` 后才允许排队。
- 打印预览页不得在没有本次拍摄照片时展示演示照片或提交打印；空状态必须引导顾客返回拍照，模板占位只能用于确认版式。
- 打印预览页不得用本地计时器模拟打印成功；提交打印必须具备已上传后端的 `photo_id`、登录令牌和真实打印机选择，未上传、上传失败或非真实活动会话应明确阻止。
- 后端打印执行不得在无默认打印机、找不到打印机或打印机状态不可用时模拟完成；这些情况必须将打印任务标记为失败并记录错误原因。
- 打印预览页不得展示静态打印历史、固定今日打印量或假墨水统计；可见队列摘要必须来自真实系统打印队列。
- 打印预览页不得暴露未接入真实打印任务的纸张选择、色彩优化和内联校准控件；出纸尺寸应从当前模板只读展示，校准能力应走真实校准页或后端持久化能力。
- 打印校准后端持久化当前明确返回 501；前端不得暴露只显示“正在开发中”的校准占位页，运营入口应指向真实可执行的打印中心。
- 模板编辑器支持上传底图和装饰图片；底图适合复刻离线版/设计稿模板，顾客只需上传底图后拖放照片框定位拍照区域。
- 模板编辑器空画布必须给出上传底图、添加照片框和快速版式入口；照片框在画布上需要明显可见，避免顾客不知道下一步该做什么。
- 模板编辑器底图可锁定，避免调整照片框时误替换或移除底图；多照片框模板应能快速切换照片号。
- 模板编辑器拖动和缩放元素时应限制在画布边界内，并提供中心/边缘吸附参考线，降低自定义模板对齐成本。
- 模板编辑器画布区域应支持低摩擦缩放操作，并保持左右侧栏固定宽度，避免自定义模板编辑时画布和属性面板互相挤压。
- 模板编辑器拖动和缩放元素时应先更新临时交互布局，松手后再写入一次撤销记录，避免一次拖动生成大量 undo 步骤。
- 模板编辑器底图作为模板 layout 的背景层时可保存可选 `background.zIndex`，用于和普通元素统一排序；相关类型契约必须同步更新。
- Webhook 配置和发送日志属于团队管理数据；创建、列表、删除和日志读取必须验证当前用户已认证，删除和日志读取必须校验 webhook 所属团队成员身份。
- Stripe webhook 是公开入口，但必须配置 `STRIPE_WEBHOOK_SECRET` 并通过 Stripe 签名校验；缺少 webhook secret 时必须 fail closed。
- 后端 service 类默认按实例使用；路由不得把实例方法当静态方法调用。
